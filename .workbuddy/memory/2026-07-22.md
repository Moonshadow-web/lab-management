## 质控管理卡顿优化（用户反馈：打开/切换各功能卡顿，数据量会越来越多）
- 根因：单页 `QCList.vue` 在文件顶部**静态** `import` 了三个重子组件（`Comparison`/`InterlabComparison`/`TargetValueBoard`）+ `echarts`(1.13MB) + `exceljs`(938KB)。导致「质控管理」一打开就要下载解析 **1.3MB 的 QCList chunk**，且切换 tab 每次重新挂载重组件+重拉数据。
- 方案（低风险、纯前端、不改任何接口/数据逻辑）：
  1. 三个子组件改为 `defineAsyncComponent(() => import(...))` 懒加载——仅当切到对应 tab 才下载该组件 chunk。
  2. `echarts` 从顶部 import 改为 `renderChart` 内 `import('echarts').then(...)` 动态加载——仅打开 LJ 质控图时加载。
  3. `exceljs` 从顶部 import 改为 `exportExcel` 内 `(await import('exceljs')).default` 动态加载——仅点导出时加载。
- 验证（本地 build）：QCList chunk **1295.68 kB → 68 kB**（~19×）；`echarts` 落入懒加载 chunk `index-DgQRkS4v.js` 且**未**在 index.html 预载；`exceljs` 独立懒 chunk。子组件各自独立 chunk（Comparison 44.8k / Interlab 26.4k / TargetValueBoard 23k）。
- commit `508875b`，已 push，部署中（任务 DFDNKb）。部署后用 Playwright 冒烟：登录→点「质控管理」→依次切 仪器间比对/室间比对/累靶/LJ质控图/室间质评，断言无 pageerror/console error，并验证 echarts 仅在触发图表时才下载。
- 注意：`defineAsyncComponent` 在 tab 未激活时不渲染（与原 `v-if` 行为一致），首次切到某 tab 有 <1s 的 chunk 下载空白期，属预期。若后续要进一步消除「切换卡顿」，可考虑 `<keep-alive>` 缓存已加载 tab（避免重复挂载重拉数据），但会改变行为、需谨慎评估数据时效。
- 后端数据量增长侧：列表接口多走通用 `make_router`（含 `page`/`page_size` 分页，EQA 列表前端已传 `page_size:500`），月结 `listQCSummaries` 也带 `page_size:500`——当前分页机制已就位；后续若单表行数极大，再按需加 DB 索引/服务端游标即可（禁 DROP，ADD COLUMN/INDEX 幂等）。

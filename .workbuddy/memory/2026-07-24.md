## 微信提醒内容乱码修复（HTML → Markdown）

- **根因**：ServerChan 的 `desp` 按 **Markdown** 渲染，而 `reminder_engine.run_reminders` 微信发送处直接用了 `_render_html`（HTML）→ 微信把 `<table><tr><td>` 当纯文本显示成「乱码」。用户收到后反馈。
- **修复**：`backend/app/services/reminder_engine.py` 新增 `_render_markdown(items)`（按 🔴已逾期 / 🟡即将到期 分组、手机易读），微信发送 `desp=_render_markdown(...)` 改走 Markdown；邮件仍用 HTML。commit `3888bb5`（fix: 微信(ServerChan)提醒内容改 Markdown 渲染…），build mark `serverchan-markdown-2026-07-24`。
- **部署插曲（重要）**：首次 markdown 部署在 EKS 滚动发布时 `Liveness/Readiness probe failed: dial tcp ...:8080: connection refused` + `Back-off restarting failed container` → 部署版本失败（build mark 未切换，仍旧版）。排查：改动仅新增函数+改一个调用，py_compile 通过、函数位置正确，不可能在启动时崩溃 → 判定为 CloudBase EKS 偶发基础设施/构建问题，非代码问题。重试部署后成功（build mark 切到 `serverchan-markdown-2026-07-24`）。
  - **经验**：CloudBase cloudrun 滚动发布偶发 probe 失败可直接重试部署，不必怀疑业务代码（先 py_compile + 复查作用域确认代码无启动期问题）。
- **验证（三路全过）**：
  1. 代码部署：build mark = `serverchan-markdown-2026-07-24`。
  2. 手动 Markdown 推送：用用户贴出的 9 条原始内容重建，`render_markdown` 直接发 ServerChan → `code:0 SUCCESS`（7 逾期 + 2 即将到期，无 HTML 乱码）。
  3. **引擎自动 Markdown 推送**：`POST /reminders/run?as_of=2026-07-24`（dry_run 是 query 参数，非 form！早先用 form 传导致实际跑了真实 run、显示 planned=0 的假象）→ 真实 run 返回 `items_sent:5, wx_sent:1`，5 条 08-07 到期事项（刚进 14 天窗口）以归整 Markdown 推到微信。

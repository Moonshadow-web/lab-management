## 仪器排序「有管理人却沉底」排查（残留多 uvicorn 进程占端口）
- 现象：用户反馈有管理人的设备（如凝血分析仪）仍排在最后。TestClient（import app.main）验证 `_instrument_order` 正确（凝血在前），但**真实 HTTP 请求**排序却是 id.desc（凝血最后）。
- 根因：历史上多次用 Bash 后台启动 uvicorn，残留**多个 python/uvicorn 进程**；netstat 常只显示一个 LISTENING，但新启动的 uvicorn 因 `10048 端口已被占用` 绑定失败并退出（日志 `ERROR: [Errno 10048]`），请求一直被跑着**旧代码**的残留进程处理（旧代码 order_by 未生效 → 走 `Model.id.desc()`）。
- 修复：**杀掉所有 python 进程**（`Get-Process -Name python* | Stop-Process -Force`，本机仅后端用 python），确认 8123 监听数=0 后，干净启动 `venv/Scripts/python.exe -m uvicorn app.main:app --port 8123`（不加 --reload 更稳）。新进程真正 LISTENING 后，真实 HTTP 验证 order_by 生效（凝血分析仪索引 0-3/15 居前，首个空管理人索引 26，末尾全为空管理人）。
- 教训（重要）：①改后端后若现象"代码对但接口不对"，优先怀疑**是否有残留旧 uvicorn 进程在占端口**；②Windows 上 netstat 显示单 LISTENING 不代表只有一个 uvicorn 实例（其他可能处于 TIME_WAIT 或绑定失败前的占用）；③重启后端前务必 `Stop-Process` 全部 python，并核对启动日志**无 10048 错误** + netstat 确认新 PID 真正 LISTENING；④验证排序/过滤必须用**真实 HTTP**（urllib 登录→请求），不能只信 TestClient（TestClient 走 in-process import，不会暴露多进程端口冲突）。

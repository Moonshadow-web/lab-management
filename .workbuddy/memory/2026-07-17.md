## 室间比对 interlab - 创建计划 500 修复（关键线上故障）
- 现象：线上 POST /api/v1/interlab/plans 返回 500，日志 `no such table: interlab_items`。
- 根因：`main.py` _migrate_schema 中删表后调 `Base.metadata.create_all(bind=engine)` 另开一条连接；DROP 在 conn 未提交事务中，新连接(尤其 WAL/CFS)看不到 → 误判表仍在而跳过建表 → 事务提交后表消失 → 后续写入 500。前序部署已把 interlab_items 误删，表彻底缺失。
- 修复（两版，均已 push+deploy）：
  1. 改用 `Base.metadata.create_all(bind=conn)`（同一连接，未提交 DROP 可见）。
  2. 条件化：仅当 `interlab_items` **缺失(len(cols)==0)** 或 **含旧列(our_value/ref_value)** 才 DROP+重建；表已为新结构且存在时跳过 → 容器重启不清空已有比对数据。
- 验证：本地起服务跑通 创建→5水平结果→保存→预览→生成报告→删除；线上同样端到端通过(201→200保存值落库→报告 docx)。
- commits: 8467a9a(同连接修复) + 438dbe3(缺失/旧结构均重建) → push origin main → tcb cloudrun deploy --force。
- ⚠️ 教训：SQLite 迁移中「DROP 后 create_all」必须用同一连接，且要避免每次启动无条件 DROP（除非检测旧结构才重建），否则容器重启会丢数据。

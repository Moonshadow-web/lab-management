### 重大踩坑：自动迁移里用了 DROP COLUMN（已修正，重要教训）
- **错误做法**：在 `_migrate_schema` 的 reconcile 里用 `DROP COLUMN` 删 `interlab_items.our_value/ref_value` 等过时列，每次启动都跑。
- **用户质疑**（正确）：迁移代码绝不能用 DROP 做 schema 演进（不删列、不删索引、不删表）。线上 interlab_items 那些过时列在"本模块无正式数据"的假设下被丢弃，但假设一旦有数据就是事故。SQLite 改列正路是 `ALTER ADD COLUMN` + `UPDATE 兜底`，或对遗留列 `ALTER COLUMN DROP NOT NULL` 放宽约束（保留列、保留数据）。
- **修正后的非破坏性 reconcile**（commit 本轮最终版）：
  - 扫描 `interlab_items`/`interlab_levels`，凡列名不在模型期望集合、且 `notnull=1`，执行 `ALTER TABLE ... ALTER COLUMN col DROP NOT NULL`（SQLite ≥ 3.35 支持，Python 3.13 自带满足）。
  - **绝不 DROP COLUMN、DROP INDEX、DROP TABLE**。列与历史数据完整保留，INSERT 留空不再触发 IntegrityError。
  - 失败兜底：极旧 SQLite 不支持 `DROP NOT NULL` 时 print 错误并保留原状，由人工处理。
- **原则固化**（已写入 MEMORY.md）：
  1. 迁移里禁止任何 DROP（列/索引/表）。
  2. 新列用 `ALTER ADD COLUMN` 补；老列用 `ALTER COLUMN DROP NOT NULL` 放宽。
  3. "expected = 模型列集合" 的耦合很脆弱，模型加列时必须同步 alters + expected，禁靠 DROP 自纠。

### 副踩坑：Starlette ServerErrorMiddleware 吞掉 500 detail
- 现象：`raise HTTPException(500, detail=traceback)` 客户端拿到纯文本 "Internal Server Error" 而不是 `{"detail":...}`。Starlette 的 `ServerErrorMiddleware` 对 500 响应会替换 body 为纯文本，不调用用户 detail。

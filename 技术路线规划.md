# 生免组实验室管理系统 · 技术路线规划

> 编制依据：用户最早总指令（8 模块 + 遵循 ISO 15189 + 最终部署腾讯云开发 CloudBase）。
> 本文只讲**技术实现路线**与**管理系统建设中的关键风险**（含业务方易忽略项）；业务功能范围以 `MEMORY.md`「原始需求」为准。

## 0. 当前状态基线（已发生的事实）
| 项 | 现状 |
|---|---|
| 架构 | FastAPI + SQLAlchemy 2.0 + SQLite（后端） / Vue3 + Vite + Element Plus + Pinia（前端），本地运行 |
| 覆盖 | 8 模块基础 CRUD + EQA 自动上报；JWT 鉴权（admin/admin123） |
| 已踩坑（宝贵经验） | 双 `app.db` 权威库混淆；静态路由需排在 `/{id}` 前；**单位换算上报错误**；登录时序 401；openpyxl 缺失；`create_all` 不 ALTER，靠 `_migrate_schema` 补列 |
| 目标态 | 腾讯云开发（CloudBase）生产部署，合规、可追溯、可审计、可备份、可恢复 |

## 1. 总体技术架构（演进方向）

```
本地原型                          →   腾讯云开发生产态
┌─────────────┐                      ┌──────────────────────────┐
│ Vue3 SPA     │  静态托管            │ CloudBase 静态网站托管     │
│ (localhost)  │  ──────────►        │ (HTTPS/CDN，前端)          │
├─────────────┤                      ├──────────────────────────┤
│ FastAPI      │  云函数/云托管       │ CloudBase 云函数/云托管     │
│ uvicorn 本地 │  ──────────►        │ (FastAPI，自动扩缩)        │
├─────────────┤                      ├──────────────────────────┤
│ SQLite 单文件│  托管数据库          │ CloudBase MySQL / 文档数据库│
│ data/app.db  │  ──────────►        │ (自动备份、主从)           │
├─────────────┤                      ├──────────────────────────┤
│ 本地磁盘上传 │  云存储              │ CloudBase 云存储(对象存储)  │
│ data/uploads │  ──────────►        │ (SOP/说明书/报告文件)      │
└─────────────┘                      └──────────────────────────┘
```

**分层**：表现层（Vue SPA）→ 接入层（CloudBase Hosting + API 网关）→ 服务层（FastAPI）→ 数据层（托管数据库 + 云存储）。

## 2. 技术栈与选型依据
| 层 | 选型 | 理由 | 上云迁移点 |
|---|---|---|---|
| 前端 | Vue3 + Vite + Element Plus + Pinia | 医学后台成熟、组件全、国产环境友好 | 构建产物托管到 CloudBase Hosting；`baseURL` 改云函数域名 |
| 后端 | FastAPI（异步、类型、自动 OpenAPI） | 适合云函数、便于自动化测试 | 入口改为云函数 handler；去除本地 `--reload` |
| ORM | SQLAlchemy 2.0 | 已落地，迁移成本低 | 连接串切到云数据库；注意连接池在云函数下的复用 |
| 数据库 | **CloudBase MySQL（推荐）** | 关系型、与现有模型兼容、支持事务/约束/审计 | 替代 SQLite；需写迁移脚本 |
| 文件 | CloudBase 云存储 | 替代本地磁盘，带鉴权访问 | 上传/下载改为对象存储 SDK |
| 鉴权 | JWT + 角色 | 已用；需补 RBAC | 密钥托管到云密钥管理，禁硬编码 |

## 4. 关键技术问题与风险（重点：您易忽略但会致命）
> 这部分是规划的核心价值——管理系统不止"能增删改查"，下面任一项缺失都会在 ISO 15189 评审或真实故障中暴露。

### A. 合规与可追溯（ISO 15189 硬性要求）
- **审计追踪（audit trail）**：谁、在何时、改了哪条记录的哪个字段、改前/改后值。医学实验室认可能否通过，审计追踪是必查项。现状 `audit_logs` 仅是雏形，需做成**全表写操作自动留痕中间件**。
- **数据不可篡改/防抵赖**：关键记录（SOP 发布、质评确认、不符合项关闭）需不可被静默改写，应有版本快照 + 操作人绑定。
- **电子签名**：审核、发布、确认类操作应留电子签名/确认痕迹，而非仅"保存"。

### B. 权限与角色（RBAC，不止一个 admin）
- 现实角色至少：系统管理员、专业组长、质控员、普通检验师。需按**模块 + 操作（读/写/审）**粒度授权，且接口层强制鉴权（不能只靠前端隐藏按钮）。
- 当前 `admin/admin123` 弱口令上线即违规，必须改强口令 + 账号体系。

### C. 数据安全与等保
- 内部系统也含敏感数据（参考区间、人员信息、质评成绩），按等保 **二级起步**。
- 传输加密（HTTPS）、存储加密、账号安全、操作日志脱敏。

### D. 备份与灾备（SQLite 单文件 = 单点故障）
- 本地 `app.db` 一旦损坏/误删即全损。云上必须用**托管数据库 + 自动每日备份 + 定期恢复演练**。
- 文件（SOP/报告）同样需异地备份，不能只存本地磁盘。

### E. 数据一致性与主数据管理（已踩坑，必须系统化）
- **单位换算错误**已真实发生（内分泌上报填了仪器原值而非上报值）。需建设**单位字典 + 换算引擎**，录入即校验，上报前自动转目标单位，禁止手工拍脑袋。
- **主数据（MDM）**：检验项目、仪器、方法学名称需统一维护（现有 `aliases` 机制是雏形，要扩展为权威主数据，避免"同一项目多种叫法"导致关联断裂）。

### F. 与外部系统对接（LIS / HIS / 流水线）
- 数据从哪来？手工录入易错、滞后；对接 LIS 自动取数才准。需决策：**是否对接、对接哪些、用何协议（HL7/私有）**。
- 对接带来的新问题：接口可用性、数据同步一致性、断点续传、字段映射。

### G. 文档与版本管理
- SOP 生效/作废流程、版本追溯、作废后旧版仍可查（合规要求"留痕"）。
- 防误删：文件删除走软删除 + 回收，不走物理删除。

### H. 报表与导出合规
- 导出的 PDF 报告应带：水印、页码、生成时间、操作人（ISO 15189 对"记录"的要求）。
- 导出动作本身也需审计。

### I. 质量保障（医学系统不能出错）
- 单元/集成/端到端测试；数据校验规则（防止已遇的 `None`→422/500）。
- 输入校验：数值范围、单位、必填项，前端+后端双重校验。

### J. 性能与可用性
- 云函数冷启动、并发上限、超时；需做接口耗时与并发压测。
- 大数据量：560+ 项目、历年质评，需分页 + 索引，避免全表扫。

### K. 可维护性与演进
- 配置外置（密钥、环境变量），日志集中，监控告警（接口错误率、备份失败告警）。
- 数据库变更脚本**可回滚**（schema 演进不能只靠 `_migrate_schema` 裸补列）。

### L. 多端适配
- 科室大屏看板、手机查阅；您已有"检验项目速查小程序"（560+ 项），可考虑**同源维护 / 数据联动**，避免两套数据。

## 5. 技术实施路线（工程分期，非业务阶段）
- **A. 数据层云化**：SQLite → CloudBase 数据库；编写迁移脚本（含历史数据）；开启自动备份；文件迁云存储。
- **B. 服务治理**：RBAC 权限中间件、审计中间件（全表写留痕）、统一异常处理、输入校验、单位/主数据服务、电子签名留痕。
- **C. 前端增强**：权限路由、审计视图、导出组件（带水印）、电子签名交互、软删除回收站。
- **D. 上云部署**：FastAPI 改云函数/云托管；前端托管；CI 自动构建部署；等保加固、密钥托管。
- **E. 对接与增强**：LIS/流水线对接、报表合规水印、移动端/小程序联动、监控告警。

---

## 6. 需要您拍板的关键决策（blocking 后续路线）
1. **云数据库选型**：CloudBase MySQL（推荐，兼容现有 ORM，事务/约束完整） vs 文档型（需改造 ORM，关系查询弱）。
2. **是否对接 LIS / 流水线自动取数**：决定数据准确性上限与对接工作量。
3. **账号体系**：科室统一账号 / 微信扫码登录 / 沿用 admin（需强化口令）。
4. **电子签名与等保要求**：ISO 15189 是否要求电子签名留痕？等保级别目标？
5. **移动端**：是否需要小程序/手机端联动（与现有速查小程序同源）？

> 以上决策确定后，技术路线即可锁定，后续按 A→E 分期实施。
